## PART A — DISCOVERY SUMMARY

### 1. Current Architecture Summary

HybridTurtle is a Next.js 14 + TypeScript + Prisma + SQLite trading system with:

- **7-stage scan pipeline** (universe → tech filters → classify → rank → risk gates → anti-chase → size)
- **Dual scoring** (BQS/FWS/NCS composite 0–100)
- **4 risk profiles** (CONSERVATIVE, BALANCED, SMALL_ACCOUNT, AGGRESSIVE)
- **Day-based execution modes** (PLANNING, BLOCKED, PLANNED, OPPORTUNISTIC)
- **6 hard risk gates** (open risk, max positions, sleeve cap, cluster cap, sector cap, position size cap)
- **Monotonic stop management** (INITIAL → BREAKEVEN → LOCK_08R → LOCK_1R_TRAIL)
- **Multi-signal regime detection** (SPY + VWRL dual benchmark, stability, chop band)
- **14-phase prediction engine** (all advisory-only: conformal, failure modes, stress test, MI, ridge, lead-lag, GNN, Bayesian, Kelly, Meta-RL, TradePulse, causal/IRM, VPIN, sentiment)
- **Trading 212 integration** (dual account ISA/Invest, execution pipeline, stop sync, account fallback)
- **Evidence framework** (EV tracker, candidate outcomes, forward enrichment, filter scorecard, score validation, trade logger, execution audit, execution drag)
- **Automation** (nightly 9-step, midday sync, watchdog, heartbeat, 16-point health check, Telegram alerts, DB backup)
- **Dashboard** with today-directive, plan page, scan page, risk page, portfolio page, settings, notifications, onboarding

### 3. Operating Modes Currently Present

**No formal operating mode system exists.** The system has:

- **Day-based execution mode** (`execution-mode.ts`): PLANNING / BLOCKED / PLANNED / OPPORTUNISTIC — but this is day-of-week logic, not a user-selectable operating mode
- **Kill switch** in settings (binary on/off)
- **Auto-trading toggle** (binary on/off)
- No BEGINNER / NORMAL / AGGRESSIVE_QUALITY / CAPITAL_PRESERVATION / RECOVERY mode

This is the single largest architectural gap.

4. **Stale data entries** — Data freshness checked but not enforced at execution point

### 5. Over-Engineering Risks

1. **14-phase prediction engine** — Comprehensive but most modules have small sample sizes; risk of overfitting to noise
2. **GNN (Phase 8)** — Complex graph neural network with limited training data
3. **Meta-RL (Phase 11)** — Reinforcement learning with sparse reward signal
4. **Causal IRM (Phase 14)** — Invariant risk minimisation adds complexity with unproven benefit
5. **Sentiment fusion** — News/analyst/short interest with limited data quality for UK/EU stocks

All prediction modules are correctly bounded as advisory-only, which mitigates the over-engineering risk. The system should not add more prediction complexity until existing modules are validated with sufficient trade samples.

### 8. Where the System May Miss Winners

1. **ext_atr = 0.8 threshold** — May be too aggressive, blocking valid breakouts that gap slightly
2. **Weekend no-entry block** — Valid weekend triggers wait until the next weekday session
3. **WAIT_PULLBACK too strict** — Pullback zone of ±0.25 ATR is tight; many pullbacks overshoot
4. **FWS > 65 Auto-No** — May block candidates with high BQS that have recoverable weakness
5. **3-day regime stability requirement** — Delays entry during regime transitions
6. **No partial position entry** — Cannot scale in with a starter position

### 9. Where the System May Buy Too Late

1. **Entry trigger = 20d high + 0.1×ATR** — By definition, buying after the breakout
2. **Tuesday planned execution** — 2 days after Sunday scan; price may have moved
3. **Anti-chase gap guard** — Correct behaviour but may need tuning based on evidence
4. **No intraday trigger monitoring** — All triggers evaluated at close prices
5. **Scan runs nightly** — Intraday breakouts are missed until evening

### 10. Where the System May Hold Losers Too Long

1. **INITIAL stop at entry - 1R** — Held until 1.5R before breakeven; can sit at -0.5R for weeks
2. **No dead-money timer** — No rule for "if position hasn't moved in N days, review"
3. **No opportunity cost comparison** — Dead position blocks capital for stronger candidate
4. **No trend deterioration detection** — Position can weaken without triggering exit review
5. **Laggard detection exists but has no action engine** — Detection without recommendation

### 11. Where the UI May Confuse a Beginner

2. **Risk gates displayed as pass/fail codes** — G1–G6 means nothing to a beginner
3. **BQS/FWS/NCS scores** — Numbers without explanation

## PART C — PHASE 2: 10/10 OPERATING MODEL

### Observable Behaviour

"After this change, the system will output a single daily directive combining risk profile, operating mode, market conditions, system health, and all safety gates — answering 'What should I do today?' with a clear, actionable, beginner-safe answer."

### Decision Tree

```
INPUT: riskProfile + operatingMode + dayOfWeek + regime + health + riskBudget
       + positions + stops + candidates + antiChase + earnings + dataFreshness
       + t212Sync + executionQuality + equity + heartbeat + backup

HARD GATES (always win):
  IF kill switch active           → SYSTEM_BLOCKED
  IF emergency do-nothing         → SYSTEM_BLOCKED
  IF health = RED                 → SYSTEM_BLOCKED
  IF heartbeat > 18h stale        → SYSTEM_BLOCKED
  IF equity ≤ 0                   → SYSTEM_BLOCKED
  IF backup required but missing  → SYSTEM_BLOCKED
  IF t212 not connected           → BUY_BLOCKED
  IF data stale (critical)        → BUY_BLOCKED
  IF regime = BEARISH             → BUY_BLOCKED
  IF regime = SIDEWAYS            → BUY_BLOCKED (soft: WATCH_CANDIDATES allowed)

OPERATING MODE OVERRIDES:
  IF mode = CAPITAL_PRESERVATION  → CAPITAL_PRESERVATION_ACTIVE (exits only)
  IF mode = RECOVERY              → RECOVERY_ACTIVE (A-grade only, reduced sizing)
  IF mode = RESEARCH              → RESEARCH_ONLY (no execution)

NORMAL FLOW:
  IF Monday                       → MANAGE_EXISTING or UPDATE_STOPS
  IF Sunday/Saturday              → PREPARE_PLAN
  IF laggards detected            → EXIT_REVIEW_REQUIRED
  IF stops pending                → UPDATE_STOPS
  IF A-grade candidates exist AND budget allows AND mode allows
                                  → BUY_ALLOWED
  IF candidates exist but B-grade → WATCH_CANDIDATES
  IF no candidates                → NO_ACTION or MANAGE_EXISTING
```

INPUTS: currentPrice, entryTrigger, atr, dayOfWeek, slippageBuffer,

IF price < trigger:

IF price ≥ noChase:

IF operating mode = strict:
